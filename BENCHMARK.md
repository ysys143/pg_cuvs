# pg_cuvs Benchmark

> **The headline is not "GPU is faster."** It is *where the time goes once the GPU
> removes distance computation.* When CAGRA collapses the per-query distance work to
> a ~0.7 ms GPU kernel, the bottleneck moves to **IPC round-trip and PostgreSQL heap
> access** — not the math. This document characterizes that shift with a latency
> decomposition, then backs it with a pgvector comparison on real embeddings and a
> multi-tenant filtered-search selectivity sweep.

All numbers here are VM-measured and traceable to a source artifact:

| Source | What it holds |
|--------|---------------|
| [`docs/profiling-results.md`](docs/profiling-results.md) | Search / build / export latency decomposition (Nsight Systems + perf) |
| [`design/BENCHMARK_CROSSOVER.md`](design/BENCHMARK_CROSSOVER.md) | Full crossover methodology + synthetic pilot + 50M competitive + Cohere real-embedding run |
| [`bench/results/*.csv`](bench/results/) | Raw per-engine result rows |
| [`docs/filter-threshold-experiment.md`](docs/filter-threshold-experiment.md) | Selectivity × correlation sweep for filtered BF |

Reproduction harness: [`bench/`](bench/) (see [§5](#5-reproduce-it-yourself)).

## 1. Overhead characterization (the core result)

### 1.1 Search path — where a single query's time goes

CAGRA search, Q=1, 1M × 1024, daemon wall-clock = `pg_stat_gpu_search.avg_latency_us`,
GPU breakdown from Nsight Systems `cuda_gpu_kern_sum` / `cuda_gpu_mem_time_sum`.

| Component | Time (Q=1) | Share | Source |
|-----------|-----------:|------:|--------|
| **GPU kernel** (CAGRA search + topk) | **~715 µs** | **66%** | `cuda_gpu_kern_sum` |
| └ `multi_cta_search::search_kernel` | 698 µs | 65% | 304 instances |
| **memcpy** (H2D query + D2H result) | **~4.4 µs** | **0.4%** | `cuda_gpu_mem_time_sum` |
| **IPC + overhead** (shm read/write, socket, result marshalling) | **~358 µs** | **33%** | wall − kernel − memcpy |
| **Daemon wall-clock (total)** | **1077.5 µs** | 100% | `avg_latency_us` |

**Reading it:**

- **GPU kernel : IPC overhead ≈ 2 : 1.** CAGRA search is **GPU-bound** — the kernel is
  66% of the wall-clock. This is the opposite of CPU HNSW, where the distance math *is*
  the cost.
- **memcpy is 0.4% — negligible.** The zero-copy shared-memory design transfers only
  Q×K (query in, TID+distance out), not the corpus. The 4 GB one-time index upload
  (876 ms) is a lazy-load that happens once per index, not per query.
- **IPC is now the second-largest line item (33%).** Once distance computation is gone,
  the cost of *reaching* the GPU (shm handshake + socket wakeup + result marshalling)
  is what's left to optimize — not the search itself.
- **Batching amortizes it.** A Q=100 batch kernel runs in ~1.27 ms (max) — ~1.8× the
  Q=1 kernel for 100× the queries. Throughput scales far better than single-query
  latency suggests.

### 1.2 Build path — the GPU floor and the removable PG overhead

`CREATE INDEX USING cagra`, 1M × 1024, EXTENDED storage. GPU build segment from daemon
journal timestamps (`handle_build` → `built index`); backend from `task-clock` perf.

| Component | Time | Share | Source |
|-----------|-----:|------:|--------|
| **GPU CAGRA build** (shm read + H2D + graph construction) | **~68 s** | **82%** | daemon journal |
| **PG backend** (heap scan + detoast + memcpy + shm write) | **~15.5 s** | **18%** | total − GPU build |
| **Build wall-clock (total)** | **83.5 s** | 100% | `\timing` |

The GPU build (~68 s) is a **floor** — it is cuVS-internal and shared with a raw cuVS
call; no amount of PostgreSQL-side work can push below it. The backend ~15.5 s is the
**only removable part**, and it is *entirely* PG-integration overhead. The north-star
metric is therefore **"how much of the PG overhead over raw cuVS do we remove,"** not
the wall-clock ratio.

Three levers, all measured on N=500K × 1024:

| Lever | Backend overhead | Source |
|-------|-----------------:|--------|
| Baseline (heap + double memcpy) | — | — |
| **memfd handoff** (ADR-057) — daemon mmaps the corpus, zero copy | ~6.3 s, **peak RSS −32%** | profiling §7 |
| **+ parallel maintenance workers** (ADR-058) — distributed heap scan | ~4.5 s (−27%) | profiling §8 |
| **+ daemon multi-partial direct H2D** (ADR-059) — removes the leader merge copy | **~3.7 s** | profiling §9 |
| **PLAIN storage** (ADR-043) — removes detoast | backend 15.5 s → 8.7 s | profiling §4 |

memfd also makes crash-orphan leak **structurally impossible** (anonymous fd, no
`/dev/shm` name to leak) — validated across a SIGKILL/SIGSEGV/OOM-killer matrix.

**Net:** combining PLAIN + memfd + parallel workers shrinks the backend from ~15.5 s to
**~2–4 s**, i.e. build wall-clock 83.5 s → ~70–72 s = **GPU floor + minimal dispatch ≈
~95% of a raw cuVS call**, while keeping PostgreSQL MVCC, durability, and DDL
integration. The build wall-clock improvement is marginal *because the GPU floor
dominates* — the value is the removed backend overhead, not the clock.

> This **corrected an earlier code-analysis estimate** (ADR-034) that assumed "GPU build
> ~10 s vs PG overhead ~45 s." The measurement inverted it: GPU build dominates at 82%.

### 1.3 Storage: TOAST (EXTENDED) vs PLAIN

Same 1M × 1024 data in both layouts.

| Metric | EXTENDED | PLAIN | Delta |
|--------|---------:|------:|-------|
| CAGRA build time | 83.5 s | 76.7 s | PLAIN **8.1% faster** |
| backend segment (build − GPU 68 s) | ~15.5 s | ~8.7 s | detoast ≈ 6.8 s |
| main heap size | 58 MB | 7813 MB | PLAIN 134× larger |
| INSERT throughput (100K) | 3130 ms | 2811 ms | PLAIN 10% faster |
| search latency | storage-independent | storage-independent | CAGRA graph is VRAM-resident; heap recheck touches only K rows |

PLAIN's build saving is **~8%**, not the ~25–35% an earlier estimate assumed — because
detoast is only ~8% of the build and 82% is the (storage-independent) GPU floor. PLAIN
still wins on build, INSERT, and total disk, at the cost of a 134× larger main heap
(hurts non-vector scans) — hence it remains a *vector-only-table* recommendation.

### 1.4 Export path (`USING pg_cuvs_hnsw`) — the buffer-manager wall

Converting CAGRA → pgvector HNSW writes 1M pages. Backend CPU breakdown (perf):

| Component | Share of backend CPU | Note |
|-----------|---------------------:|------|
| `write_elem_page` | 77% | per-page write body |
| └ **buffer manager** (`ReadBuffer_common`, P_NEW + relation-extension lock) | **~39% total** | single largest cost |
| └ WAL (`log_newpage_buffer` → `XLogInsert`) | ~14% total | full-page image + crc32c |
| └ page fill (`PageAddItem` / memcpy) | ~25% total | vector/neighbor copy |

WAL delta: 1,000,238 records, 4441 MB, 1,000,026 full-page images.

Sequential `ReadBuffer(P_NEW)` serializes on the relation-extension lock, which is why
parallel page-write is blocked — this **quantitatively confirmed** the decision to defer
parallel export (ADR-035). `UNLOGGED` removes the ~14% WAL slice as a practical
mitigation.

## 2. Comparison vs pgvector (secondary)

The comparison matters, but it is downstream of §1: pg_cuvs wins where the GPU's
flat-latency floor beats CPU HNSW's per-query growth, and loses where IPC overhead
exceeds the (small) search it replaces.

### 2.1 Real embeddings — Cohere Wikipedia 1M × 1024 (cosine)

Single A100-40GB, k=10, iso-data, exact brute-force ground truth.
Raw: [`bench/results/cohere_N1000000_summary.csv`](bench/results/cohere_N1000000_summary.csv).

| System | recall@10 | QPS | p50 | Params |
|--------|----------:|----:|----:|--------|
| **pg_cuvs CAGRA** (GPU search) | **0.9912** | 227 | **4.4 ms** | k=100 |
| pgvector HNSW | 0.9891 | 45 | 22 ms | ef_search=400 |
| pgvector HNSW | 0.9392 | 130 | 7.6 ms | ef_search=80 |
| pgvector IVFFlat | 0.9766 | 8.6 | 115 ms | probes=128 |
| pg_cuvs `import_hnsw` (3I) | **0.9993** | 16.9 | 61 ms | ef=512 |
| cagra-hnsw-cpu (cuVS lib, CPU search) | 0.9975 | 611¹ | 12 ms | ef=512 |

¹ 12-core batch QPS; single-query p50 = 12 ms.

At equivalent recall (~0.99), pg_cuvs CAGRA delivers **5 ms p50 vs pgvector HNSW's
22 ms** — and the gap is the §1.1 story: CAGRA's search is a fixed GPU kernel, while
HNSW must walk more of the graph (higher `ef_search`) to reach 0.99 on 1024-dim real
embeddings.

**Build (1M × 1024):**

| Path | Build time | Note |
|------|-----------:|------|
| pgvector HNSW native | 285 s | m=16, ef=64, 16 GB mem, 7 workers |
| CAGRA build + `import_hnsw` (3I) | **142 s** | **2.0× faster**, then served by pgvector HNSW |
| cagra-hnsw (cuVS lib, GPU build only) | 12 s | no PostgreSQL COPY overhead |

### 2.2 Synthetic crossover pilot — where the line is

Single A100, k=10, clustered synthetic, iso-recall target 0.95, concurrency=8.
Full table in [`BENCHMARK_CROSSOVER.md` §11](design/BENCHMARK_CROSSOVER.md).

| N | dim | engine | build (s) | p50 (µs) | QPS (c=8) | recall@10 |
|--:|----:|--------|----------:|---------:|----------:|----------:|
| 1,000 | 384 | HNSW | 0.25 | **224** | **24,513** | 0.988 |
| 1,000 | 384 | CAGRA | 0.14 | 871 | 1,864 | 1.000 |
| 100,000 | 384 | HNSW | 25.5 | 8,232 | 876 | 0.932 |
| 100,000 | 384 | CAGRA | **2.77** | **1,228** | **1,206** | 0.982 |
| 1,000,000 | 1536 | HNSW | 2,721 | 14,969 | 400 | 0.910 |
| 1,000,000 | 1536 | CAGRA | **75.3** | **1,702** | **893** | 0.995 |

**CAGRA latency is near-flat in N** (871 → 1,228 µs from 1K → 100K at dim 384), while
HNSW grows steeply (224 → 8,232 µs). The **latency crossover is ≈ N 10K–100K**; build
advantage widens with both N and dim (9× → 36× at 1M × 1536). Below ~10K, pgvector HNSW
wins on every axis — the IPC round-trip (§1.1) is not worth paying for a tiny search.

### 2.3 50M × 384 competitive (4-way) — the honest ceiling

2×A100-40GB, clustered synthetic. Full writeup: [§12](design/BENCHMARK_CROSSOVER.md).

| engine | build | QPS | p50 | recall@10 | note |
|--------|------:|----:|----:|----------:|------|
| pgvector HNSW | 21,879 s | 546 | 13 ms | (not re-measured) | 97 GB index, disk spill |
| VectorChord (vchordrq) | 5,784 s | 152 | 49 ms | 0.9991 | IVF excels on clustered synthetic |
| pgvectorscale DiskANN | TIMEOUT (>5 h) | — | — | — | 2 GB cache → graph degrades at ~0.9% load |
| **pg_cuvs CAGRA** | **FAILED** | — | — | — | 73 GiB raw fp32 > 2×40 GB VRAM incl. build workspace |

**This is the single-node VRAM ceiling, stated plainly.** 50M × 384 fp32 = 73 GiB of
raw vectors; with CAGRA's build workspace it does not fit in 2×40 GB. The honest
recommendation: for billion-scale / larger-than-VRAM, use pgvectorscale or VectorChord
(see README "When to Use"). pg_cuvs targets the **VRAM-resident, query-cost-dominated**
segment (online RAG, multi-tenant), not raw scale.

## 3. Multi-tenant filtered search — selectivity sweep

Supports the filtered brute-force path (D-wedge post-filter + 3O BITSET pre-filter), the
multi-tenant primitive: each tenant queries with a TID filter over its own rows.
N=200K × 128, uniform random, k=10, overfetch=4, 5 reps/cell.
Full table: [`docs/filter-threshold-experiment.md`](docs/filter-threshold-experiment.md).

| selectivity | random recall | mixed recall | spatial recall | med latency |
|------------:|:-------------:|:------------:|:--------------:|------------:|
| unfiltered  | — | — | 1.00 | 1.97 ms |
| 1%  (n=2k)  | 0.20 | 1.00 | 1.00 | **1.32 ms** |
| 5%  (n=10k) | 0.80 | 1.00 | 1.00 | 1.47 ms |
| 10% (n=20k) | 1.00 | 1.00 | 1.00 | 1.65 ms |
| 25% (n=50k) | 1.00 | 1.00 | 1.00 | 2.10 ms |
| 50% (n=100k)| 1.00 | 1.00 | 1.00 | 2.82 ms |

**Two findings:**

1. **Filtered search at low selectivity is *faster* than unfiltered** (~33% at sel=1%) —
   the daemon searches a smaller candidate space. The crossover (filtered = unfiltered
   latency) is ≈ sel 15%; above that, the filter membership test costs more than it saves.
2. **D-wedge post-filter recall holds at sel ≥ 10% for all correlations.** It only drops
   on the worst-case *random* column below 5% (0.80 at 5%, 0.20 at 1%) — because the
   k×4 overfetch pool runs out of in-filter rows. Real multi-tenant workloads have
   spatial correlation (tenants query near their own data), where recall stays 1.00 even
   at 1%. This is exactly why `cuvs.filter_auto_threshold = 0.05` switches to the 3O
   BITSET pre-filter below 5% selectivity.

## 4. Honest limitations

- **Most pilot/crossover data is synthetic.** Clustered synthetic flatters IVF
  (VectorChord) and gives optimistic recall; uniform-random gives recall=1.0 that real
  high-dim embeddings won't match (distance concentration). The §2.1 Cohere run is the
  one real-embedding anchor; treat synthetic crossover *coordinates* as
  distribution-dependent.
- **Single-GPU QPS ceiling ≈ 1K (small/medium N).** A single daemon's per-query IPC
  (§1.1) caps single-GPU throughput; multi-GPU sharding raises it but adds merge latency
  (recall +13%, latency +70% at shard_count=2 on 100K).
- **pg_cuvs is not a WAL-logged mutable native index** — comparisons assume a
  static/batch-built index. Streaming writes go through `cuvsCagraExtend` (3Q) or the
  `.delta` path, characterized separately ([profiling §10](docs/profiling-results.md)).
- **`pg_stat_gpu_search.p50` is a log2 bucket** — for engine-to-engine latency we use
  client-side `\timing` / `avg_latency_us`, not the bucketed percentile.
- **Profiling hardware is a GCP VM without PMU counters** — see §0.

## 5. Reproduce it yourself

Run on a GPU VM with the daemon up (`IDX_DIR` must match the daemon `--index-dir`). The
harness keeps pgvector and pg_cuvs on the **same `vector` column and same SQL**
(`ORDER BY v <-> q LIMIT k`); only the index definition differs (`USING hnsw` vs
`USING cagra`), so the measured code path is shared. **iso-recall**: each engine's knob
(`hnsw.ef_search` / `cuvs.k`) is swept to the minimum value meeting `RECALL_TARGET`
before comparing latency/QPS.

```bash
# 1) generate data + exact ground truth (numpy env)
python3 bench/gen_dataset.py --n 1000000 --dim 1024 --queries 1000 --dist random --out bench/data
python3 bench/gt.py --data bench/data --k 100

# 2) pgvector HNSW
ENGINE=hnsw  N=1000000 DIM=1024 K=10 RECALL_TARGET=0.95 bash bench/run_pilot.sh
# 3) pg_cuvs CAGRA (daemon required)
ENGINE=cagra N=1000000 DIM=1024 K=10 RECALL_TARGET=0.95 IDX_DIR=/tmp/cuvs_indexes bash bench/run_pilot.sh

cat bench/results/pilot.csv
```
